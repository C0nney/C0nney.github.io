浅谈Frida注入原理及绕过思路

发表于 2024-11-12
字数统计: 2.2k 字 | 阅读时长 ≈ 9 分钟

浅谈Frida注入原理及绕过思路，并附上一个简单的实例。

打算尝试系统梳理下常见的Frida检测原理及绕过手段，目前先举例定位pthread_create。

Frida的综合注入机制

Gadget注入

  • Gadget模式：Frida Gadget 是一个静态库，可以预先打包到目标应用中。

    • 目标应用启动时，Gadget库会自动加载，并建立与Frida的通信通道。
    • 这避免了动态注入的检测风险，但需要提前修改目标APK。

动态注入

  • 在运行时通过 ptrace 暂停目标进程。
  • 使用 dlopen 动态加载 libfrida-gadget.so 或其他自定义的动态库。
  • 目标库加载后，通过 Inline Hook 实现对目标函数的劫持。

两种方法的配合

  • ptrace 用于初始注入和代码植入。
  • Inline Hook 用于劫持关键函数，实现动态分析和修改。

Ptrace Hook

什么是 **ptrace**？

ptrace 是一个 Linux 提供的系统调用，允许一个进程（父进程）监视和控制另一个进程（子进程）的执行。它常被用于调试和修改目标进程的内存、寄存器等。

Frida中如何使用 **ptrace**？

Frida 使用 ptrace 作为其一种注入手段，尤其是在以下情况下：

  • 远程注入：通过 frida-server，Frida 使用 ptrace 将自己的动态库注入到目标进程中。
  • 挂起进程：Frida 使用 ptrace 暂停目标进程，以便将其动态库（如 libfrida-gadget.so）注入到目标进程的地址空间。
  • 内存写入：通过 ptrace，Frida 可以将自定义代码注入到目标进程的内存中，并修改目标进程的指令指针来执行注入的代码。
优点
  • 不需要修改目标应用的二进制文件。
  • 适用于运行时注入和调试。
缺点
  • 容易被检测：ptrace 的使用在某些情况下会触发应用的反调试逻辑。

Inline Hook

什么是 Inline Hook？

Inline Hook 是一种通过直接修改目标进程的内存（通常是代码段中的函数起始部分）来实现函数劫持的方法。其核心原理是：

  • 修改函数开头的指令：将原始函数的前几条指令替换为跳转（jmp）指令，跳转到自定义的代码（通常是自己的Hook函数）。
  • 执行自定义逻辑后返回：自定义逻辑执行完成后，可以跳回原函数继续执行。

Frida中的Inline Hook

Frida 在目标函数上实现Hook时，使用了Inline Hook技术。例如：

  • Hook Android或iOS的系统API（如 open、read 等）。
  • Hook Native层（C/C++）函数，通过拦截函数的执行流程来分析和修改目标程序的行为。
Frida的实现方式
  • ARM/ARM64架构：修改目标函数的前几条指令（如 BL、MOV 等），将其跳转到自定义的Thunk代码。
  • x86/x86_64架构：修改 jmp 或 call 指令的目标地址，将执行流程重定向到Frida的Hook逻辑。
优点
  • 性能较高，几乎无中断感。
  • 非常灵活，可以直接修改目标函数的执行逻辑。
缺点
  • 对于某些具有自校验功能的应用（如代码段完整性校验），Inline Hook 可能会被检测。

常见Frida检测

ptrace占坑、进程名检测、端口检测（绕过简单）

D-Bus通信协议检测：App向每个端口都发送D-Bus认证消息，然后利用strcmp()或strstr()函数检测回复消息。

maps、fd检测：

1.检测frida特征（hluda-server绕过：生成随机的.so名，去除frida特征）

2.但有的会检测maps和fd中是否存在/data/local/tmp/，甚至只存在tmp字段app就执行kill。

（那就重定向，直接hook open函数，将原程序的maps文件中带有tmp的行过滤掉，剩余内容输出到另一个文件中，然后修改open的返回值，指向新生成的文件）

线程检测（定位到调用pthread_create的函数，nop或者替换）

我个人简单以两个主要函数作为节点：dlopen(android_dlopen_ext)和libc

hook-dlopen

动态链接库是通过系统的 dlopen进行加载的，在 android 7.0 之后使用的是 android_dlopen_ext。dlopen 是 linker 的高级接口，linker提供核心功能，dlopen 负责调用。

.init_proc 和 .ini_array 执行的时机很早，且执行过一次之后就不再执行。因为我们需要在 android_dlopen_ext 运行过程中hook。

检测原理

  • 应用可能会调用 dlopen 或相关函数（如 dlsym 和 dlclose）来检查特定库（如 libfrida-gadget.so）是否被加载。

hook**dlopen**主要是用于定位注入点。

因此需要先hook dlopen定位检测点哪个.so中。

并且，根据so的加载流程，linker->init_proc函数->.init_array中的函数->JNI_OnLoad函数，需要进一步确定检测点的时机。

此时需要先hook JNI_Onload定位检测点是否在JNI_Onload之前。

如果检测在.init_xx函数，由于这个时候处于dlopen调用后，init_xx已经执行完成了，想要再往前hook linker的call_function并不容易。

此时，可以考虑在.init_xx中找调用时机比较早的外部函数的位置。

hook-libc

找到注入点后，就可以对具体的frida检测点下手了。这就轮到libc出场了。

libc 是C语言的标准库，提供了许多基础函数，例如 open、read、write、 mmap、fork、exit、_exit、kill、 等。它们被广泛用于系统调用和文件操作。

检测原理

  • 文件检测：应用可能通过调用 open 来检查 /proc/self/maps 文件，以判断是否有与Frida相关的库被加载。
  • 内存检测：应用可能通过 mmap 或 mprotect 检查内存段的权限和内容。
  • 进程检测：通过调用 fork 或类似的函数创建新进程，用来验证是否存在调试环境。

网上对frida的检测通常会使用openat、open、strstr、pthread_create、snprintf、sprintf、readlinkat等一系列函数。

1.C++基础

  • c++是c的超集，任何合法的 C 程序都是合法的 C++ 程序。
  • 大多数的C++编译器并不在乎源文件的扩展名，如未使用拓展名默认为.cpp，其它的文件名也可以是.h文件。
  • c++是面向对象语言。

标准的 C++ 由三个重要部分组成：

  1. 核心语言，提供了所有构件块，包括变量、数据类型和常量，等等。
  2. C++ 标准库，提供了大量的函数，用于操作文件、字符串等。
  3. 标准模板库（STL），提供了大量的方法，用于操作数据结构等。

约定俗称，.h文件主要负责类成员变量和方法的声明；.cpp文件主要负责成员变量和方法的定义。（类比java中.class和.java）

在这段代码中，delete a; 的作用是释放之前通过 new 操作符动态分配的内存。

详细解释如下：

  1. **动态内存分配 (**new 操作符): 当代码执行到 A *a = new A(1,2); 时，会动态地在堆上为对象 A 分配内存，并调用构造函数 A(int, int) 初始化这个新创建的对象。这里，a 是一个指向该动态分配对象的指针。
  2. 使用动态分配的对象: 接下来的 a->sum(3); 调用了对象 a 的 sum 方法，传入参数 3。这里，-> 运算符用于通过指针访问对象的方法或属性。
  3. **释放内存 (**delete 操作符): 当执行到 delete a; 时，C++ 会调用该对象的析构函数（如果有的话），然后释放之前为该对象分配的内存。这对于避免内存泄漏非常重要，内存泄漏是指程序未能释放不再使用的内存，随着时间推移会导致程序占用的内存持续增长。
  4. 自动内存管理: 代码中还展示了另一种内存管理方式，即自动分配（栈上分配）内存，A a2 = A(1,2);。这种方式下，对象 a2 的生命周期与声明它的作用域相关联，当离开该作用域（在这个例子中是 main 函数结束时），编译器会自动调用析构函数并释放内存，无需手动调用 delete。

总结来说，delete a; 是手动管理动态内存的重要步骤，确保了不会因忘记释放内存而导致的资源泄露问题。不过，现代C++编程更倾向于使用智能指针（如 std::unique_ptr 或 std::shared_ptr）来自动管理动态内存，从而减少手动使用 new 和 delete 的需求，进一步提升代码的安全性和可靠性。

指针

指针：是一个变量，这个变量的值是另一个变量的内存地址。也就是说，指针是一个指向内存地址的变量。
Java 中没有指针的概念的，但是其实 Java 中除了基本数据类，大部分情况下使用都是指针。

* ：有两个作用：

i. 用于定义一个指针： type *var_name; ，var_name 是一个指针变量，如 int *p;

ii. 用于对一个指针取内容： *var_name， 如 *p 的值是 1。

& ：是一个取址符号

其用于获取一个变量所在的内存地址。如 &a; 的值是 a 所在内存的位置，即 a 的地址。

new与delete

指针变量可以通过new动态创建。

注意：在C++中，使用new操作符动态分配内存时，所分配的内存是在堆上进行的。与在栈上创建的对象不同，堆上的内存不会随着作用域的结束而自动回收。需要我们自己手动回收内存，否则将会导致内存泄漏。回收内存则是通过delete关键字进行的。

纯虚函数

在 Java 中，我们经常会使用 interface 或 abstract 来定义一些接口，方便代码规范和拓展，但是在 C++ 没有这样的方法，但是可以有类似的实现，那就是：纯虚函数。

有虚函数的类如A是一个抽象类，不能被直接定义使用。

2.JNI开发

JNI原理

Java语言的执行环境是Java虚拟机(JVM)，JVM其实是主机环境中的一个进程，每个JVM虚拟机都在本地环境中有一个JavaVM结构体，该结构体在创建JVM虚拟机时被返回。JNI全局仅仅有一个，JavaVM是Java虚拟机在JNI层的代表，一个JVM对应一个JavaVM结构。

JNI提供了多种函数来处理不同的数据类型。在与Java交互时，需要注意类型转换和管理。以下是一些常见的数据类型和它们的转换方法：

Java基本类型和对应的JNI类型

Java 类型 JNI 类型 转换函数
boolean jboolean env->CallBooleanMethod
byte jbyte env->CallByteMethod
char jchar env->CallCharMethod
short jshort env->CallShortMethod
int jint env->CallIntMethod
long jlong env->CallLongMethod
float jfloat env->CallFloatMethod
double jdouble env->CallDoubleMethod

3.使用ndk-build编译静/动态链接库、可执行文件

文件结构

项目文件下必需jni文件，jni下要有Android.mk

  • LOCAL_PATH := $(call my-dir) 不用改，表示当前目录，便于LOCAL_SRC_FILES可以使用相对路径
  • 包含了NDK内置的**$(CLEAR_VARS)脚本，这个脚本会清除所有以前定义的LOCAL_**变量。这样可以确保每次定义一个新的模块时，不会受到前一个模块设置的影响。
  • LOCAL_MODULE := 【】 编译后的模块名，默认在后面加上.so
  • LOCAL_SRC_FILES 指定要编译的源文件，如有多个源文件用空格/Tab分隔。必须同一行指定，如要换行使用\区分
  • LOCAL_CFLAGS := -fPIE Android 5.0 及更高版本要求执行的可执行文件必须是位置无关的可执行文件（Position Independent Executables，简称 PIE）。这是 Android 系统为了提高安全性而引入的要求。所以编译时需要加上这句。
  • LOCAL_LDLIBS += -L$(SYSROOT)/usr/lib -llog -llog 表示链接 liblog.so 库，该库是Android平台上用于日志记录的库，由于我的代码中用到了liblog库，所以需要链接，否则无法编译。如果缺少可能会出现以下报错：

ptraceInject.o: In function log_message:

ptraceInject.c:5: undefined reference to __android_log_print

collect2: error: ld returned 1 exit status

  • include $(BUILD_EXECUTABLE) 告诉构建系统该模块应被构建成一个可执行文件。

    • $(BUILD_SHARED_LIBRARY) 构建共享库（动态链接库.so）
    • $(BUILD_STATIC_LIBRARY) 构建静态链接库（.a 文件）
    • $(BUILD_PREBUILT) 包含已经预编译的库或二进制文件。

obj文件夹包含中间文件，这些文件用于存放编译过程中生成的对象文件（.o 文件）和其他中间数据。这些文件夹主要用于构建过程，通常不会部署到设备上。这些文件帮助构建系统跟踪构建过程，包括依赖关系和构建状态。
